# 문서 작성 가이드 (RAG 친화 정보구조)

이 문서는 사이트에 빌드되지 않는 내부 기준 문서입니다 (`_` 프리픽스). 사람이 직접
쓰는 문서와 `blog-v2`에서 자동 생성되는 문서가 **모두 이 틀을 따라야** 합니다.

> 영어 원본: `_authoring.md`. 영어가 source of truth이며, 한쪽을 고치면 다른 쪽도
> 같은 턴에 반영합니다.

## 두 영역

이 저장소는 두 영역을 빌드합니다.

- **API** (`/api`) — blog-v2 백엔드 엔드포인트 모음. OpenAPI에서 Scalar로 생성됩니다.
- **Docs** (사이드바) — **mogumogu.dev 블로그 사이트 사용 설명서**. 각 페이지가
  무엇이고, 화면의 버튼·요소가 무슨 역할을 하는지 설명합니다.

이 가이드는 **Docs 영역**의 작성 기준입니다.

## 왜 이 틀이 필요한가

Docs는 사람이 읽는 **사용 설명서**이자 검색·질의응답에 쓰이는 **RAG 소스**를
겸합니다. 청크 자기완결성과 메타데이터는 *작성 시점*에 결정되므로, 문서를 다 쓴 뒤
RAG에 맞추면 전면 재작성이 됩니다. 처음부터 이 틀로 쓰면 작성한 문서가 그대로 RAG에
적재됩니다.

## 6대 원칙

1. **1 문서 = 1 페이지.** 한 문서는 블로그의 한 화면(페이지)을 설명하고, 그 페이지의
   모든 영역·버튼의 역할을 담습니다. 단독으로 읽어도 완결됩니다.
2. **헤딩 = 청크 경계.** H2/H3가 검색 청크 단위가 됩니다. 각 H2는 그 자체로 읽혀야
   합니다.
3. **메타데이터를 frontmatter로.** 아래 스키마의 필드를 빠짐없이 채웁니다 — 검색·필터·
   신선도 신호가 여기서 나옵니다.
4. **섹션은 질문형/작업형으로.** 사용자의 실제 질문 임베딩과 정렬되도록 "어떻게 …
   하나요?" 또는 "… 하기" 형태를 선호합니다.
5. **텍스트 우선, 이미지엔 alt 필수.** RAG는 이미지를 읽지 못합니다. 핵심 정보는 본문
   텍스트에 두고, 모든 이미지에 설명적 alt를 답니다.
6. **3개국어 동일 구조.** en/ko/ja가 같은 헤딩·같은 명시 앵커(`{#id}`)를 공유해야
   링크와 청크가 어긋나지 않습니다.

## Frontmatter 스키마

| 필드 | 필수 | 용도 | 예시 |
| --- | :---: | --- | --- |
| `title` | ✅ | 청크 제목. H1과 일치 | `게시글 목록` |
| `description` | ✅ | 한 문장 요약(≤120자). 청크 요약 겸 meta description | `게시글 목록 화면이 무엇을 보여주고 글을 어떻게 여는지` |
| `keywords` | ✅ | 검색·임베딩 매칭 키워드. 동의어 포함 | `[posts, list, feed]` |
| `tags` | ✅ | 패싯 필터링용 분류 | `[posts, navigation]` |
| `source` | ✅ | 출처: `manual` 또는 `generated` | `manual` |
| `last_verified` | ✅ | 신선도 신호 (YYYY-MM-DD) | `2026-06-20` |
| `audience` | ⬜ | 독자: `reader` / `admin` / `all` | `reader` |
| `sidebar_position` | ⬜ | 사이드바 순서 | `2` |

자동 생성 문서는 반드시 `source: generated`로 표시해, 사람이 쓴 문서와 구분되고
검증·갱신 대상을 가려낼 수 있게 합니다.

## 헤딩 규칙 (청크 경계)

- **H1은 문서당 하나** — 페이지 그 자체이며 `title`과 일치합니다.
- **H2는 주요 섹션 = 1차 청크 경계** — 각 H2는 단독으로 읽혀야 합니다.
- **H3는 하위 항목** — 레벨을 건너뛰지 않습니다 (H2 → H4 금지).
- **대명사 대신 주어를 반복** — "이것/그것"은 앞 청크에 의존하므로, 청크가 잘리면
  맥락이 사라집니다. 주어를 다시 적습니다.
- **너무 긴 섹션은 분할** — 한 섹션이 화면 한 장을 크게 넘기면 H3로 쪼갭니다.

## 카테고리 골격

Docs 사이드바는 **블로그 사이트의 사이드바 메뉴 구조를 순서대로, 2뎁스까지
미러링**합니다. 상위 메뉴 = 카테고리, 그 안의 페이지 = 문서. 문서 하나가 블로그
페이지 하나에 대응합니다.

```
docs/docs/
  <상위메뉴-1>/
    _category_.json
    <페이지-a>.md     # 블로그의 한 페이지 = 문서 하나
    <페이지-b>.md
  <상위메뉴-2>/
    _category_.json
    ...
```

(실제 메뉴 이름·순서는 블로그 사이드바를 그대로 따릅니다.)

각 카테고리에는 `_category_.json`을 둡니다.

```json
{
  "label": "<상위메뉴 이름>",
  "position": 2,
  "link": { "type": "generated-index", "slug": "/docs/<menu-slug>" }
}
```

- `label` — 블로그 사이드바 메뉴명 그대로
- `position` — 사이드바 순서 (블로그 메뉴 순서와 일치)
- `link.slug` — 항상 `/docs/<menu-slug>` 형태로 고정 (생성 인덱스가 `/category/*`로
  새는 불일치를 막습니다)

## 콘텐츠 파이프라인 (매핑 → 채우기)

CI/CD가 두 단계로 Docs를 유지합니다. 두 단계 사이의 계약이 **매핑 파일**
(`docs-map.yaml`)이며, 이 파일이 사이드바 구조의 source of truth입니다.

1. **매핑 생성·갱신** — 블로그 사이드바를 읽어 메뉴·페이지 구조를 `docs-map.yaml`에
   선언합니다. 블로그 메뉴가 바뀌면 이 파일이 갱신됩니다.
2. **콘텐츠 채우기** — 매핑의 각 페이지에 대해 실제 블로그 페이지(`url`)를 보고,
   위 frontmatter 스키마 + `_template.md` 구조에 맞춰 en/ko/ja 문서를 생성합니다.
   `status`로 채움 상태를 추적해, 매핑이 갱신되면 새 페이지만 채웁니다.

매핑 스키마 (`docs-map.yaml`):

```yaml
menus:                      # 사이드바 상위 메뉴 = 카테고리 (순서대로)
  - slug: posts             # /docs/posts, 카테고리 디렉터리명
    label: { en: Posts, ko: 게시글, ja: 記事 }
    position: 1
    pages:                  # 메뉴 안의 페이지 = 문서 (2뎁스)
      - slug: list
        label: { en: Post list, ko: 게시글 목록, ja: 記事一覧 }
        url: /              # 블로그에서 이 페이지의 실제 경로 (2단계가 방문)
        status: pending     # pending | generated | manual
```

| 필드 | 레벨 | 용도 |
| --- | --- | --- |
| `menus[].slug` | 메뉴 | 카테고리 디렉터리명·slug |
| `menus[].label` | 메뉴 | 3개국어 카테고리 라벨 |
| `menus[].position` | 메뉴 | 사이드바 순서 |
| `pages[].slug` | 페이지 | 문서 파일명·slug |
| `pages[].label` | 페이지 | 3개국어 페이지 라벨 |
| `pages[].url` | 페이지 | 블로그 실제 경로 (2단계가 보고 채움) |
| `pages[].status` | 페이지 | `pending` / `generated` / `manual` |

## 작성 체크리스트

- [ ] 1 문서 = 1 페이지, 단독으로 읽어도 완결되는가
- [ ] frontmatter 필수 필드를 모두 채웠는가
- [ ] 각 H2가 자기완결적인가 (대명사로 앞 섹션에 의존하지 않는가)
- [ ] 섹션 헤딩이 질문형/작업형인가
- [ ] 모든 이미지에 설명적 alt, 핵심 정보는 본문 텍스트에도 있는가
- [ ] en/ko/ja가 동일 헤딩·동일 명시 앵커를 쓰는가
- [ ] 자동 생성 문서라면 `source: generated`로 표시했는가
